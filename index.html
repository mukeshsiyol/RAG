01

What is RAG? The Core Idea

Imagine you have a very smart friend who has read thousands of books — but those books are from 2 years ago. If you ask them "What happened in yesterday's cricket match?", they'd have no idea. Now imagine giving that friend a newspaper from today right before you ask the question. They read the newspaper, and now they can answer perfectly.

That is exactly what Retrieval-Augmented Generation (RAG) does for AI language models.

💡 Simple Definition

RAG = Retrieval (find relevant info) + Augmented (add it to the question) + Generation (AI writes the answer using that info). It lets an AI answer questions using documents it has never been trained on.

RAG was introduced in a 2020 paper by Facebook AI Research (now Meta AI). The key insight was: instead of trying to bake all knowledge into a model's weights (which is expensive and goes stale), let the model look things up at query time.

The Three Actors in RAG

🔍

The Retriever

Searches a knowledge base (your documents) and finds the most relevant pieces of text for the user's question.

📚

The Knowledge Base

Your collection of documents: PDFs, web pages, internal wikis, databases — anything the model should know about.

🤖

The Generator (LLM)

Takes the user's question + the retrieved documents and generates a well-formed, accurate, grounded answer.

02

Why RAG Exists – The Problem with Pure LLMs

Large Language Models (LLMs) like GPT-4 or Gemini are trained on massive amounts of internet text. But this training process has fundamental limitations.

The 4 Core Problems RAG Solves

❌ Problem 1: Knowledge Cutoff

Every LLM has a "training cutoff" date. After that date, it knows nothing. If your model was trained with data up to January 2024, it cannot answer questions about events in 2025 — no matter how you ask.

❌ Problem 2: Hallucination

When an LLM doesn't know something, it often makes up a confident-sounding but completely wrong answer. This is called "hallucination" and is dangerous in production systems (legal, medical, finance).

❌ Problem 3: Private / Proprietary Data

Your company's internal documents, customer data, product manuals — none of that was in the LLM's training data. The model has zero knowledge of your specific context.

❌ Problem 4: Retraining is Expensive

Training a large model costs millions of dollars. You can't retrain every time your documents are updated. You need a way to give the model fresh information cheaply and instantly.

✅

RAG solves all four problems

By retrieving real documents at query time, RAG gives the LLM fresh, private, and verified information — dramatically reducing hallucinations and eliminating the need for retraining.

03

Full RAG Architecture

RAG has two major phases. Think of them as preparation and question-answering.

Complete RAG System Architecture
📌 Key Insight

The Offline Phase (indexing) happens once when you set up the system. The Online Phase (retrieval + generation) happens every time a user asks a question — in real time, usually in under 2 seconds.

04

Indexing Pipeline — The Offline Phase

Before RAG can answer any question, you must prepare your documents. This is called the indexing pipeline. It converts raw text into a searchable, numerical format. This happens once (and again whenever your documents are updated).

Indexing Pipeline — Step by Step
📄

Step 1: Document Loading

Collect all your source documents. These can be PDF files, Word documents, web pages scraped from the internet, database records, Notion pages, Confluence articles, GitHub README files — anything textual. Libraries like LangChain and LlamaIndex have built-in loaders for dozens of formats.

✂️

Step 2: Chunking (Text Splitting)

A 50-page PDF cannot be fed into an LLM all at once. We split documents into smaller chunks — typically 200 to 500 words each, with some overlap (e.g. last 50 words of chunk N appear at the start of chunk N+1). Overlap ensures context is not lost at boundaries. Common strategies: fixed-size splits, paragraph splits, semantic splits.

🔢

Step 3: Embedding (Text → Vectors)

Each chunk is passed through an embedding model (e.g. OpenAI text-embedding-3-small, Sentence-BERT, Cohere embed). The model converts the text into a vector — a list of numbers (e.g. 1536 numbers) that represent the meaning of the text in mathematical space. Two chunks with similar meanings will have vectors that are numerically close together.

// Example: Text becomes a vector
"What is the return policy?"
→ embed()
→ [0.12, -0.45, 0.89, 0.02, -0.33, ...] // 1536 numbers
🗃️

Step 4: Storing in a Vector Database

The vectors (and the original text chunks they represent) are stored in a vector database. This is a special kind of database optimised for similarity search — finding which stored vectors are closest to a new query vector. Popular options: Pinecone, Weaviate, Chroma, FAISS, Qdrant, Milvus.

🏷️

Step 5: Metadata Tagging (Optional but Important)

Along with the vector, you store metadata: the document title, page number, date, author, category tag, etc. This lets you apply filters during retrieval — e.g. "only search documents from the last 6 months" or "only search from the HR policy category". This dramatically improves precision.

Similarity: How Does the Search Work?

The vector database compares the query vector against all stored document vectors using a mathematical formula called cosine similarity. It finds the stored vectors that point in the most "similar direction" in vector space. This is like measuring the angle between two arrows — the smaller the angle, the more similar the meaning.

🧮 Cosine Similarity Intuition

If query = "password reset" and chunk = "forgot password help" — their vectors will point in nearly the same direction → high similarity score (close to 1.0). A chunk about "invoice payment" will point in a very different direction → low score (close to 0). The database returns the top-K highest-scoring chunks.

What is ANN (Approximate Nearest Neighbor) Search?

If you have 10 million document chunks, comparing your query against all 10 million one-by-one would be too slow. ANN algorithms use clever indexing (like HNSW — Hierarchical Navigable Small World graphs, or IVF — Inverted File Index) to find the top-K closest vectors in milliseconds, sacrificing a tiny bit of accuracy for massive speed gains.

Re-ranking (Optional but Recommended)

After vector search returns top-K chunks, a cross-encoder re-ranker model (like Cohere Rerank or a BERT-based model) re-scores each chunk against the query more accurately but slowly. This two-stage approach (fast vector search → slow but accurate re-ranking) gives excellent precision without sacrificing speed.

06

The Generation Step — Building the Prompt

Once you have the top chunks from retrieval, you combine them with the user's question into a single prompt and send it to the LLM. This is called prompt construction or context stuffing.

What a RAG Prompt Looks Like

# System Prompt
You are a helpful assistant. Answer the user's question using ONLY
the context provided below. If the answer is not in the context,
say "I don't know". Do not make up information.

Context:
--- Document 1: HR Policy Manual, Page 12 ---
Employees can reset their password by visiting the IT portal at
it.company.com and clicking "Forgot Password". The reset link expires
in 15 minutes.

--- Document 2: FAQ Article, Last Updated 2024-03 ---
Common password issues include: expired passwords (rotate every 90 days),
locked accounts (3 failed attempts), and forgotten passwords.

--- End of Context ---

# User Question
How do I reset my password?
✅ What the LLM Does With This

The LLM reads the context, understands the question, and synthesizes a natural-language answer using only the information provided. It might say: "To reset your password, visit it.company.com and click 'Forgot Password'. The reset link will expire after 15 minutes." — factual, grounded, no hallucination.

Key Prompt Engineering Choices

🔒

Grounding Instruction

Tell the LLM to use ONLY the provided context. This prevents it from using its internal training data (which might be wrong or stale).

🚫

Refuse Gracefully

Tell the LLM to say "I don't know" if the answer isn't in the context. This is how RAG eliminates hallucinations.

📎

Source Citations

Instruct the LLM to cite which document/page it used. This makes answers verifiable and builds user trust.

📏

Context Window Limit

LLMs have a token limit (e.g. 128K tokens). You must ensure your chunks + question fit within this window.

Vector Databases Explained

The vector database is the heart of a RAG system. It's what makes fast semantic search possible.

🧠

Think of it like this

A regular database stores text and finds it by exact keyword match — like using Ctrl+F. A vector database understands meaning — it finds text that means the same thing even if the words are completely different.

1. Query Rewriting

Users often ask vague or poorly-worded questions. Before searching, use an LLM to rewrite the query into a cleaner, more searchable form. Example: "my account is messed up" → "how to fix account access or login issues".

2. HyDE (Hypothetical Document Embeddings)

Instead of embedding the short user query (which may not match document style), use an LLM to generate a hypothetical answer, then embed that instead. This gives a richer vector that better matches how real answers are phrased in your documents.

3. Hybrid Search

Combine vector search (semantic meaning) with BM25 keyword search (exact word matching). This handles cases where vector search misses exact proper nouns, codes, or technical terms. Weaviate and Qdrant support this natively.

4. Re-ranking with Cross-Encoders

After getting top-20 results from fast vector search, a cross-encoder model reads each result alongside the query holistically (not as separate embeddings) and scores them with much higher accuracy. You keep only the top 3-5 after re-ranking.

5. Parent-Child Retrieval (Small-to-Big)

Index small chunks (50 words) for precise matching, but when you retrieve a match, return the larger parent chunk (300 words) to give the LLM more context. Best of both worlds: precise retrieval + rich context.

6. Agentic / Self-RAG

In advanced setups, the LLM itself decides when to retrieve (it might have enough information already), evaluates whether retrieved documents are relevant, and can trigger multiple retrieval rounds if the first pass wasn't sufficient.

RAG vs Fine-Tuning — When to Use What?

Both RAG and fine-tuning give LLMs access to new knowledge, but they work very differently. This is one of the most important architectural decisions in applied AI.

Aspect RAG Fine-Tuning
Cost Low — no model training needed High — GPU hours, compute cost
Update frequency Real-time — just update the DB Slow — must retrain each update
Hallucination control Strong — uses cited documents Weaker — knowledge baked in opaquely
Source attribution Easy — can cite source document Difficult — knowledge is in weights
Best for: factual Q&A ✅ Excellent Mediocre
Best for: style/tone Not great ✅ Excellent
Best for: reasoning tasks Depends on retrieval quality Better if fine-tuned on reasoning
Private / fresh data ✅ Perfect fit Data must be in training set
💡 Practical Rule of Thumb

Use RAG when the model needs access to specific facts, documents, or data that can change. Use Fine-tuning when you want the model to change HOW it behaves, talks, reasons, or formats responses. For most enterprise applications, RAG first — only fine-tune if RAG alone isn't sufficient.

Can You Use Both Together?

Absolutely — and this is often the best approach. Fine-tune the model to understand your domain's language, terminology, and response style, then add RAG on top to supply it with current, specific factual information. This is called RAG + Fine-Tuning and gives you domain expertise + up-to-date knowledge.
